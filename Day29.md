> **階段四｜怎麼落地：從技術棧示範**

![把二十八天收斂成一份可勾選的檢核表](https://raw.githubusercontent.com/nickchen1998/ithelp-2026-ai-security/main/%E5%9C%96%E6%AA%94/Day29/Day29-01-checklist-hero.png)

## 二十八天，收斂成一張表

這個系列走到今天，已經累積了很多東西：前三個階段講清楚了「法規要求什麼、標準怎麼導入、台灣有哪些評測機構」，第四階段（Day 21–28）則把這些抽象要求，一天一天地變成了真正跑得起來的程式——資料治理、輸入防禦、輸出防護、存取控制、紅隊測試、稽核日誌、供應鏈治理。

但這裡有一個現實問題：**這些散落在二十八天裡的知識與程式，別人（你的主管、稽核員、採購方、認證機構）要怎麼「一眼看懂你到底做了沒、做全了沒」？** 總不能叫他們把二十八篇文章從頭讀一遍。

所以整個系列的核心交付物，不是任何一段程式，而是今天要產出的這份東西——**《AI 專案資安合規檢核表》**。它把二十八天的所有內容收斂成一張可勾選的表：每一項該檢查什麼、對應到哪條法規原則、哪個標準控制、哪項評測、用什麼技術落地、又該怎麼驗證。這張表，就是本系列白皮書的核心。

## 為什麼需要一份檢核表？

在動手做這張表之前，先想清楚它的價值。一份好的檢核表，把「安全」這件模糊的事，變成四種具體的能力：

![檢核表把安全變成四種可操作的能力](https://raw.githubusercontent.com/nickchen1998/ithelp-2026-ai-security/main/%E5%9C%96%E6%AA%94/Day29/Day29-02-why-checklist.png)

- **可勾選**：把「我覺得應該安全」變成「這 10 項，我逐項檢查過」。清單化，才不會漏掉。
- **可對映**：每一項都同時接上法規、標準、評測，讓「合規」不再是各說各話——工程師看到技術控制、法務看到法條、稽核看到標準，講的是同一件事。
- **可驗證**：每一項都寫明「怎麼驗證、證據是什麼」，讓「有做」變成「拿得出證明」。這正是前面每一天結尾都在強調的「留下證據」。
- **可自評**：可以拿它給自己的專案打分數、找出還沒補齊的缺口——這也接回 Day 20 談的「把制度變成採購與自評的語言」。

一句話：**檢核表讓「安全」從一種感覺，變成一件可以被檢查、被證明、被交付的事。**

## 檢核表的骨架：AIEC 十大評測項目

這張表要用什麼當骨架（欄位）？答案在 Day 18 就埋好了伏筆——**AI 產品與系統評測中心（Artificial Intelligence Evaluation Center，以下簡稱 AIEC）的十大評測項目**。

用它當骨架有兩個好處。其一，它是**台灣官方**用來評測 AI 的十個面向（準確性、可靠性、彈性、安全性、資安、隱私、透明性、可解釋性、當責性、公平性），本土、權威、涵蓋完整。其二，這十項幾乎能一一對回人工智慧基本法第 4 條的七大原則（Day 8），往下又能接到 ISO/IEC 42001（人工智慧管理系統標準，Day 9–13）的控制——**它天生就站在「法規、標準、評測」三者的交會點上**，是最理想的收斂骨架。

於是每一項檢核，就長成一條完整的「從法條到程式碼」的鏈：

![一項檢核的五段鏈：評測項→原則→標準→技術→驗證](https://raw.githubusercontent.com/nickchen1998/ithelp-2026-ai-security/main/%E5%9C%96%E6%AA%94/Day29/Day29-03-five-link-chain.png)

> **AIEC 評測項 → 對應基本法原則 → ISO/IEC 42001 落點 → 技術控制（本系列哪一天） → 怎麼驗證（證據）**

## 動手：把檢核表變成可執行的自評工具

一張表格是靜態的。但如果把它寫成**機器可讀的結構化資料**，它就能變成一個可以「跑」的自評工具——拿一個專案的落實狀態去對照，自動算出準備度、列出缺口（完整檔在 `程式碼/Day29/compliance_checklist.py`）。

### 檢核表本體：結構化資料

先把上面那張表，一項一項寫成程式能讀的資料結構：

```python
# 檢核表本體：以 AIEC 十大評測項目為骨架（機器可讀）
# 每一項：AIEC 評測項、對應基本法原則、ISO/IEC 42001 落點、技術控制（本系列日次）、怎麼驗證
CHECKLIST = [
    {"id": "C1", "aiec": "資安", "principle": "資安與安全",
     "iso": "附錄 A 安全相關控制", "controls": "輸入注入防禦(D23)、存取控制(D25)、供應鏈(D28)",
     "verify": "紅隊 ASR=0%(D26)、AI-BOM 完整性驗證(D28)"},
    {"id": "C4", "aiec": "隱私", "principle": "隱私保護與資料治理",
     "iso": "附錄 A 資料相關控制", "controls": "去識別化與最小化(D22)、出口遮蔽(D24)、存取控制(D25)",
     "verify": "治理後知識庫查無個資、跨租戶被擋(D22/24/25)"},
    {"id": "C9", "aiec": "當責性", "principle": "問責",
     "iso": "附錄 A 紀錄與事件管理相關控制", "controls": "稽核日誌雜湊鏈(D27)",
     "verify": "竄改偵測（斷鏈報警）(D27)"},
    # …（C2、C3、C5–C8、C10 結構相同，完整十項見程式碼檔）
]
```

每一項就是一個字典，欄位對應到表格的五欄。這裡只列出三項作代表（其餘七項結構完全相同）。**把檢核表寫成資料而非文件，是關鍵一步**——它從此可以被程式讀取、比對、計分，而不只是給人看的文字。

### 自評：對照專案的落實狀態

接著，記錄「某個專案，這十項各自做到什麼程度」。狀態分三級：已落實、部分、尚未：

```python
# 某個 AI 專案的自評狀態：done=已落實、partial=部分、todo=尚未
# 這裡填入「本系列 RAG 醫院客服 demo」的實際落實狀態作為示範。
ASSESSMENT = {
    "C1": "done", "C2": "done", "C3": "done", "C4": "done", "C5": "done",
    "C6": "done", "C7": "done", "C8": "done", "C9": "done",
    "C10": "partial",   # 資料治理有做，但尚未做正式的偏誤（bias）量測
}

_SCORE = {"done": 1.0, "partial": 0.5, "todo": 0.0}
```

我們誠實地替本系列的 RAG 醫院客服 demo 做了自評：C1 到 C9 都在前面幾天實作並驗證過了，標記為「已落實」；但 C10 公平性，我們只做了資料源頭治理（Day 22），**還沒有做正式的偏誤（bias）量測**，所以誠實標記為「部分」。這種「不是全綠」的自評，才是真實的——檢核表的價值，正在於它能逼出這種「還沒補齊」的缺口。

### 計分與缺口分析

最後是計分邏輯——把落實狀態換算成準備度，並列出還沒補齊的項目：

```python
def assess(assessment):
    """拿專案的落實狀態對照檢核表，回傳 (準備度百分比, 缺口清單)。"""
    total = len(CHECKLIST)
    score = sum(_SCORE[assessment.get(c["id"], "todo")] for c in CHECKLIST)
    gaps = [c for c in CHECKLIST if assessment.get(c["id"], "todo") != "done"]
    return score / total * 100, gaps
```

`assess()` 把每一項的分數（已落實 1 分、部分 0.5 分、尚未 0 分）加總，除以總項數，得到一個「合規準備度」百分比；同時把所有「不是已落實」的項目挑出來，成為一份**缺口清單**——這正是專案下一步該補的功課。

最後，主程式把「印出完整檢核表」與「產出自評報告」兩件事串起來：

```python
if __name__ == "__main__":
    render_checklist()          # 印出完整檢核表（白皮書核心交付物）
    render_report(ASSESSMENT)   # 依 ASSESSMENT 印出自評報告：逐項狀態、準備度、缺口
```

其中 `render_checklist()` 與 `render_report()` 是負責格式化輸出的輔助函式（完整程式在 `compliance_checklist.py`），它們的產出就是下一節的實跑結果。

## 這份檢核表，可以怎麼用？

這張表和這支工具，不是寫完就束之高閣的作業，它有很實際的用途：

![檢核表的四種實際用途](https://raw.githubusercontent.com/nickchen1998/ithelp-2026-ai-security/main/%E5%9C%96%E6%AA%94/Day29/Day29-06-usage.png)

- **開發自評**：專案上線前，逐項自評，把準備度和缺口清單當成上線檢查表。
- **採購與驗收**：把它放進標案的規格書或驗收標準（回指 Day 20），要求廠商逐項提出佐證——這就是把合規變成採購語言。
- **送測與認證**：因為骨架就是 AIEC 十大評測項目，這份自評能直接對接 AIEC 的評測、以及 ISO/IEC 42001 的稽核，讓「送測」不再從零準備。
- **稽核證據**：每一項的「佐證」欄，配合前面各天留下的紅隊報告、稽核日誌、AI-BOM，就是稽核時能直接拿出來的證據包。

## 誠實的限制：檢核表是地板，不是天花板

一如既往，把話說清楚。這份檢核表很有用，但它有幾個必須認清的邊界：

- **勾滿不等於絕對安全**：檢核表是「最低保證」，是地板不是天花板。十項全綠，只代表「這些面向都顧到了」，不代表「沒有任何風險」——新型攻擊、沒被列進表裡的風險，它管不到。這和 Day 26 說的「ASR 0% 不等於絕對安全」是同一個道理。
- **「有勾」不等於「做對」**：把 C1 標成「已落實」很容易，但那個防禦是否真的有效，還是要靠前面每一天強調的**驗證**（紅隊、實跑證據）來支撐。檢核表逼你「提出佐證」，正是為了防止淪為紙上作業。
- **檢核表要隨環境更新**：法規會修訂、標準會改版、威脅會演化、AIEC 評測項目也可能調整。這份表不是刻在石頭上的，而要**定期複查與更新**——它是一份活的文件。

所以檢核表的正確用法，是把它當成「持續改善的起點」：定期自評、盯著缺口、隨環境更新，而不是「勾完一次就結案」。

- 二十八天散落的知識與程式，**收斂成一張可勾選、可對映、可驗證、可自評的表**，讓「安全」從一種感覺變成能被檢查與交付的事；
- 以 **AIEC 十大評測項目**為骨架，因為它天生站在「法規、標準、評測」的交會點；每一項都拉成「評測項→基本法原則→ISO/IEC 42001 落點→技術控制→怎麼驗證」的完整鏈；
- 把檢核表寫成**機器可讀的結構化資料**，就成了可執行的自評工具：實跑替本系列 demo 算出**合規準備度 95%**，並誠實列出「公平性偏誤測試待補」的缺口；
- 它可用於**開發自評、採購驗收、送測認證、稽核證據**四種場景；但它是**地板不是天花板**——勾滿不等於絕對安全、有勾不等於做對，且要隨環境持續更新。

**明天（Day 30）是這趟旅程的最後一站——總結：白皮書合成與後續追蹤。** 我們會把三十天的內容做一次總收束，說明這份檢核表與白皮書可以怎麼再利用到真實的醫院案與政府標案，並談談 AI 治理「邊做邊更新」的下一步。從 Day 1 的一條法規縫隙，到今天這份可執行的檢核表，這條「從法條到程式碼」的路，明天就要走到終點。
