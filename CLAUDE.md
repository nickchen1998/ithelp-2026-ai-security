# CLAUDE.md — 專案撰寫規範

本檔是本專案（ITHome 鐵人賽系列《從法條到程式碼：台灣 AI 治理與資安合規實戰指南》）的作業規範。撰寫任何文章草稿前先讀這份，逐條遵守。

## 專案概要

- **主題**：把上層法規標準（AI 基本法、ISO/IEC 42001 與台灣 CNS 42001、EU AI Act、NIST AI RMF、台灣評測驗證體系）一路翻譯到下層可執行的技術控制與程式碼。
- **參賽組別**：AI Security。法規標準是骨架、技術資安是血肉，兩者比例抓五五；前段先立攻防技術味，避免被當成法律讀書心得。
- **30 天大綱**：見 [README.md](README.md)。四階段——威脅與風險（Day 1–5）、制度與標準（Day 6–14）、機構與資源（Day 15–20）、技術落地（Day 21–30）。42001 逐條款深挖佔 Day 9–13。
- **最終產出**：合成一份《AI 專案資安合規檢核表 / 白皮書》，可再利用於醫院案與政府標案。

## 檔案與目錄結構

- **`DayNN.md`（根目錄）**：30 篇文章（`Day01.md`–`Day30.md`）。**檔案直接從 `# Day NN：標題` 開始，不要放 YAML frontmatter（`--- ... ---`）**——那會被 GitHub／ITHome 渲染成表格，正式文章不需要。成品要能整份直接貼上 ITHome 發布。撰稿時直接編輯對應檔案。

- **`條文/`**：陸續蒐集的條文與素材（如《人工智慧基本法》、OWASP 等 PDF），僅供本機查閱與寫作對照。**屬單人參考用途；受著作權的原文不得轉貼進文章，也不應隨公開版本（如 GitHub）散布**（見著作權紅線）。

## 著作權紅線：公開發表，從嚴處理

**本系列文章會公開發表在 ITHome（公開網頁），所以一律採最嚴格標準。** 核心判準不變：**著作權保護「表達（文字）」，不保護「概念、事實、方法、系統」。** 用自己的話講它的「意思」＝安全；搬它的「字」＝危險。

依來源把「可不可以引用原文」分成三層，落筆前先判斷手上的素材屬於哪一層：

### 第一層：ISO / CNS 標準 → 綠燈 only，不逐字引用

ISO 與 CNS 是**受著作權的私權文件**（購買僅取得「閱讀與實施」授權，不含重製散布）：

- ISO 官方版權政策明訂：未經書面許可，禁止以任何形式（**含網路公開**）複製、儲存或傳輸標準的任何部分（[ISO Copyright](https://www.iso.org/privacy-and-copyright.html)）。
- CNS 著作權屬經濟部標準檢驗局（[BSMI](https://www.bsmi.gov.tw/)），僅供內部參考／教學對照；**在公開網頁直接貼出條文段落，即使短句，仍有極高侵權風險。**

因此對 ISO/CNS，本系列**只走綠燈**——**逐字引用（含短句、含自行翻譯）一律視同紅燈**：

- ✅ 用**自己的話**解釋條款要求什麼、為什麼重要。
- ✅ 討論**架構**：章節編號與標題屬事實（如「主體條款 4–10」「附錄 A 有哪幾類控制」），可寫。
- ✅ 加上**原創詮釋、範例、對映表**（把某條控制對映到自建 RAG 做法）——最有價值也最安全。
- ✅ 使用官方繁中**術語／名詞**（單一詞彙非受保護的表達），確保用語一致。
- ✅ 一律**註明出處與版本**（如「依 ISO/IEC 42001:2023 第 6 章」）。引用出處本身不侵權。

### 第二層：法律條文 → 可自由引用（如《人工智慧基本法》）

台灣《著作權法》§9 明定憲法、法律、命令、公文**不得為著作權標的**。所以**法律條文（如《人工智慧基本法》全文、施行細則、各部會作用法）可直接引用、全文轉錄**，仍建議註明條號與出處。Day 6–8 基本法章節可放心引用條文原文。

### 第三層：開放／公開授權來源 → 依授權引用、改作

要「引用原文」時，用這些，不要用 ISO/CNS：

- **OWASP Top 10 for LLM Applications 2025**：文件明訂採 **創用 CC 姓名標示-相同方式分享 4.0（CC BY-SA 4.0）**，可自由分享與改作（含商業用途）。兩個條件：**姓名標示**（標註 OWASP、附授權連結、若有修改要註明）＋**相同方式分享（ShareAlike）**（若「改作」其內容，該衍生部分須以同一 CC BY-SA 4.0 授權釋出）。實務判斷：用**自己的話**寫十大風險＝不觸發 ShareAlike（概念／事實不受授權拘束）；**直接引用或翻譯**其原文段落＝屬改作，需標註授權並注意 ShareAlike。建議標註：「改寫自 OWASP Top 10 for LLM Applications 2025，CC BY-SA 4.0」＋連結。
- **政府公開資料**：國科會／數位部官方導引與摘要、AIEC 公開評測項目說明、政府新聞稿與公開簡報——引用並註明出處。

### 🔴 一律禁止

- 逐字照抄或整段翻譯 **ISO/CNS** 條文（尤其附錄 A 控制清單原文，最受保護）。
- 貼出／轉發購買的 **PDF、標準內表格與圖**，或把 `條文/` 內受版權檔案提交進公開版控。
- 分享標準檔案（單人使用授權）。

### ✅ 落筆前自我檢查

> 這段素材是 ISO/CNS 嗎？是 → 只能用自己的話，不逐字引。
> 若有人光看文章就不必買標準 → 抄太多了，退回重寫。
> 若看完反而更想買正本、只是更懂怎麼用 → 過關。

## 針對 42001 章節（Day 9–13）的具體做法

- **ISO／CNS 並用**：優先引用台灣國家標準 **CNS 42001**（2026/06 發布，「資訊技術－人工智慧－管理系統」，採用 ISO/IEC 42001）的官方繁中用語與條號，並與 ISO/IEC 42001:2023 原文對照。採用程度（等同 IDT／修改 MOD）以標準封面所載為準。**CNS 同樣受著作權保護，綠燈／黃燈規範一體適用。**
- **附錄 A 控制**：用「這條控制的**目的**是 X，落地成**政策／流程／技術證據**長這樣」的改寫法，不要貼原文清單。
- **SoA、稽核流程**：講**做法與步驟**（概念），不複製標準內的範本文字。
- **佐證來源分層引用**：(1) 可引原文——國科會／數位部官方導引與摘要、AIEC 公開評測項目說明、政府新聞稿與公開簡報（開放公開資料），以及《人工智慧基本法》等法律條文（著作權法 §9）；(2) 只能綠燈轉述——CNS 42001（[CNS 線上系統](https://www.cnsonline.com.tw/) 約 NT$275、可預覽）供你讀懂、對條號、統一官方術語，但**不在公開文章逐字引用條文**（見著作權紅線第一層）。

## 讀者定位與寫作風格（每篇一體適用）

本系列公開發表，**讀者是「所有對『AI 要怎麼管、怎麼做才安全』有興趣的人」**——工程師、公務員、專案經理，乃至沒有技術背景的一般讀者、程式小白。落筆前先自問：「一個沒有背景的人讀得懂嗎？」逐條遵守：

- **篇幅要夠、逐步講清楚**：寧可長而清楚，不要短而跳步。每個概念從零說明，多用生活化類比幫非技術讀者理解。忌「行家才懂」的省略。
- **論文式名詞規範**：專有名詞**第一次出現寫全名**（中文全名＋原文全名＋縮寫），例：「大型語言模型（Large Language Model，以下簡稱 LLM）」；**第二次起才用縮寫／簡稱**。
- **起承轉合是「內在邏輯」，不是標題文字**：文章要有清楚的邏輯推進，但**絕對不可**把標題寫成「起：」「承：」「轉：」「合：」或「承一／承二」這種模板字樣。每個標題都要是**能直接發布、有實質意義**的句子（如「先看三個真實場景」「為什麼舊工具必要但不夠？」）。
- **善用標題層次與排版**：層次固定為 `#` 文章大標 → `##` 各大段 → `###` 子段，一致不跳級；每個 `##`／`###` 都自成有意義的標題，讓成品可直接貼上 ITHome 發布。適當使用**列點、表格、引用區塊、程式碼區塊**呈現內容，不要整段純文字堆疊。
- **語氣：以「論文／出版書籍」的專業筆調撰寫，不要 AI／部落格口吻**。清楚易懂與專業莊重可以並存（目標像優秀的技術科普書，而非部落格貼文）。具體禁用：
  - **對讀者喊話式**的填充句：「跟著讀就懂」「我保證…也能懂」「請放心，我不會跳步」「我知道有些讀者…沒關係」「我會帶你…」「我幫你把重點…」。
  - **浮誇口語**：「震撼教育」「劇透」「威力」「可怕之處」「請你細細體會」「你可能會心一笑」，以及過量驚嘆號與「好，我們開始」「那我們正式開始」這類開場白。
  - **過度反問**：「看到了嗎？」「對吧？」偶一為之可以，但不要每段都用。
  - 改用**陳述句**直接把事實、機制、結論講清楚；用「本文／我們／筆者」而非不斷「我幫你、我帶你」。類比與逐步說明仍要保留（那是為了好懂），但語氣要收斂、莊重。
- **詳略分層（重要）**：
  - **概念／說明性內容（尤其前段 Day 1–20 的大原則、法規、標準）→ 一律寫完整**，讓所有讀者都能讀懂，不因「後面會再講」而略過。
  - **程式實作內容（尤其 Day 21–30）→ 程式本身要寫完整可跑**，但**說明聚焦在「這段程式在做什麼、為何這樣做」**即可；**前面已充分解釋過的概念，用一兩句簡短帶過並回指日次**（如「（提示注入的原理見 Day 2）」），不重複長篇鋪陳。
  - 無論詳略，**大原則永遠要交代完整**——簡短的是「已解釋過的細節」，不是「原則」。

## 其他撰寫慣例

- 語言：繁體中文；技術名詞可中英並列。
- 時效：作用法仍在滾動更新，撰文時查有無部會新規範，「邊寫邊更新」是加分項。
- 事實查核：涉及法條條號、機構名稱、成立日期、通過時序等，落筆前再確認，勿憑記憶。
- 每篇結構建議：標題 → 為什麼重要 → 內容主體（自己的話）→ 對映到技術控制／本專案範例 → 小結與明日預告；並視需要插入 16:9 配圖（見下「文章配圖」）。

## 文章配圖

- **每篇至少 4–5 張圖（硬性要求）**：避免整篇都是字，圖片幫助各類讀者理解。可用 **`gpt-image-2`** 生成概念／流程／意象圖，或用 **matplotlib** 繪製資料型圖表（長條圖、流程、對照），兩者皆可、可混用。每張圖都要在文章對應位置插入並附繁中 alt 文字。
- **命名與存放**：一律放 `圖檔/DayNN/`，檔名 `DayNN-序號-英文簡述.png`（序號兩位、由 01 起），一律 **16:9 橫向**。

以下為 `gpt-image-2` 生圖細節：

- **模型**：`gpt-image-2`（可直接生成 16:9 尺寸，不需事後裁切）。
- **金鑰**：從專案根目錄 `.env` 讀取 `OPENAI_API_KEY`。**絕不**把金鑰值寫進任何檔案、程式碼或對話輸出。
- **流程**：呼叫 OpenAI Images API 生圖 → 取回影像 → 直接寫檔到 `圖檔/DayNN/`。
- **存放與命名**：一律放對應的 `圖檔/DayNN/`；檔名 `DayNN-序號-英文簡述.png`（序號兩位、由 01 起），例如 `Day09-01-aims-overview.png`、`Day09-02-iso-cns-map.png`。
- **比例**：一律 **16:9 橫向**，直接用 `gpt-image-2` 產出 16:9。
- **圖中文字**：
  - 以**繁體中文**為主、**字數精簡**；圖片重點在「意象、概念、流程」的表達，不要塞滿說明文字。
  - 專有名詞與縮寫可**保留英文**（如 ISO/IEC 42001、CNS 42001、AIMS、RAG、OWASP、PDCA、NIST、SoA）。
- **在文章引用**：產圖後於文章對應位置用相對路徑插入，並附繁中 alt 文字，例如 `![AIMS 與 CNS 42001 對照](圖檔/Day09/Day09-02-iso-cns-map.png)`。

參考做法（載入 `.env` 後呼叫，將 b64 解碼寫檔）：

```python
import base64
from openai import OpenAI

# 先從 .env 載入 OPENAI_API_KEY（如 python-dotenv，或 export 進環境變數）
client = OpenAI()  # 自動讀取環境變數 OPENAI_API_KEY

resp = client.images.generate(
    model="gpt-image-2",          # 直接支援 16:9
    prompt="16:9 橫向；繁體中文精簡標註、字少；表達 …… 的概念／流程；專有名詞可用英文（ISO/IEC 42001、RAG）",
    size="1792x1024",             # 16:9 橫向（尺寸 token 以 API 現行為準）
)
img = base64.b64decode(resp.data[0].b64_json)
open("圖檔/Day09/Day09-01-aims-overview.png", "wb").write(img)
```
